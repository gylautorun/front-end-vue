# 2026 年长列表滚动卡顿：Vue 3 落地方案

## 核心逻辑拆分

虚拟列表算法位于相邻的 `core` 目录，视图组件只维护滚动位置、容器尺寸和 DOM 渲染：

- `FixedHeightVirtualizer`：固定高度列表，起止索引和偏移查询均为 O(1)。
- `DynamicHeightVirtualizer`：不定高度列表，未测量项目使用预估高度，实测后使用 Fenwick Tree 修正，更新和查询均为 O(log n)。
- `VirtualRange`：两种模型共享的最小结果类型，包含 `startIndex`、`endIndex`、`offsetY` 和 `totalHeight`。

两种算法刻意分成不同 class。固定高度依赖简单除法，而动态高度依赖累计高度数据结构，统一成一个 class 会产生大量模式判断和无效状态。

动态高度组件的典型接入方式如下：

```ts
const virtualizer = new DynamicHeightVirtualizer(data.length, 80);

const range = virtualizer.getRange({
    scrollTop,
    viewportHeight,
    overscanPx: 400
});

// 列表项挂载或尺寸变化后，把真实高度写回模型。
const delta = virtualizer.updateItemHeight(itemIndex, element.offsetHeight);
if (delta !== 0) {
    // 增加一个响应式版本号，让视图重新读取 range。
}
```

如果被修正的项目位于视口上方，应给容器的 `scrollTop` 加上 `delta`，保持用户当前看到的内容不跳动。多个 `ResizeObserver` 结果应先批量收集，再在同一个 `requestAnimationFrame` 中统一写回模型。

## 结论

长列表的首选方案不是把滚动计算搬进 Worker，而是先减少工作量：固定行高、只渲染视口附近节点、滚动事件每帧最多提交一次、滚动越快则增加缓冲。页面实现位于 `index.vue`，默认演示 100,000 条数据，实际 DOM 数量保持在几十个。

只有当数据排序、过滤或布局计算本身很重时才使用 Web Worker。Worker 不能操作 DOM，而且高频发送完整高度数组会产生结构化克隆成本，可能比主线程二分查找更慢。

## 卡顿的真正来源

浏览器通常只有约 16.7ms 完成一帧。长列表常见问题是：一次创建成千上万个 DOM；每次 `scroll` 都更新响应式状态；在滚动回调里交替读取布局和写样式；图片加载或动态高度不断触发重排；列表项销毁后把交互状态一起丢失。

`will-change` 只能让合成更稳定，不能减少 Vue 更新、布局计算或 DOM 数量，因此它不是独立解决方案。

## 本页面的实现

### 1. 固定高度虚拟化

每行固定为 72px，总滚动高度是 `数据量 * 72`。根据 `scrollTop / 72` 可 O(1) 得到首个可见索引，再用 `slice` 取出视口与缓冲区数据。实际节点通过 `translate3d` 移到正确位置。

固定高度是性能和稳定性最好的选择。业务允许时，应限制摘要行数、为图片声明 `width`、`height` 或 `aspect-ratio`，将卡片变成可预测高度。

### 2. rAF 合并滚动更新

`scroll` 监听器使用 `passive`，只保存最新位置。同一帧内无论触发多少次滚动事件，都只在 `requestAnimationFrame` 中提交一次 Vue 状态。这里不使用 100ms 节流，因为它会让高速滚动时可见内容明显落后于滚动条，形成白屏。

### 3. 速度自适应缓冲

每帧根据滚动距离调整 overscan：慢速时保留 6 条以减少 DOM，快速时最多保留 28 条以降低视口追上渲染区的概率。缓冲有上限，避免触控板大幅跳动时突然创建大量节点。

### 4. 控制响应式成本

完整数据使用 `shallowRef`，Vue 不会把 100,000 个只读记录递归转换成响应式代理。滚动热路径只更新三个小数值。选择状态单独存放在 `Set`，因此节点滚出并重新创建后状态仍能恢复。

### 5. 稳定布局

页面只观察视口尺寸，不观察每一行。列表使用 `contain: strict` 隔离布局影响，并关闭滚动锚定。实际内容只改变 transform，避免在滚动热路径触发布局读取。

### 6. 为什么使用 translate3d 定位可见项目层

虚拟列表存在两个不同职责的 DOM 层：占位层负责产生完整滚动范围，可见项目层只包含当前视口附近的少量节点。可见项目层默认位于占位层顶部，因此每次渲染区间变化后，必须将它移动到 `startIndex` 对应的位置：

```text
固定高度：offsetY = startIndex * itemHeight

不定高度：offsetY = prefixHeight(startIndex)
```

页面使用：

```css
transform: translate3d(0, var(--offset-y), 0);
```

而不是频繁修改 `top` 或 `margin-top`，主要有以下原因。

#### 1. 减少布局阶段工作

`top`、`margin` 和部分尺寸属性属于布局属性。它们变化后，浏览器通常需要重新计算元素位置，并可能影响同一布局上下文中的其他元素。

`transform` 改变的是元素最终绘制位置，不改变它在正常布局中的几何占位。浏览器通常可以跳过大范围 layout，将更新集中在 paint/composite 阶段。虚拟列表滚动时每帧都可能改变偏移，因此避免重复布局很有价值。

#### 2. 更适合高频连续位移

transform 是浏览器为动画和连续移动重点优化的属性。可见项目层是一个整体，只更新父层 transform，就能移动内部几十个项目，不需要逐项修改位置。

`translate3d` 的第三个参数虽然是 0，但它明确使用三维 transform。浏览器通常会考虑将该元素提升到独立合成层，使滚动过程中主要更新图层变换，而不是重新绘制全部页面。

这里需要使用“通常”而不是“一定”：是否创建独立图层由浏览器决定，不能把 `translate3d` 当成强制 GPU 加速开关。

#### 3. 支持亚像素位置

不定高项目经过 ResizeObserver 测量后可能产生小数高度。transform 可以保留亚像素偏移，使累计高度定位更准确。代码仍会忽略小于 0.5px 的高度变化，防止测量噪声反复触发更新。

#### 4. 与绝对定位占位结构解耦

可见项目层使用绝对定位脱离普通文档流，完整滚动高度由占位层单独承担。因此 transform 只负责视觉位置，不会改变占位层高度，也不会把下面的兄弟节点推开。

#### 5. 为什么配合 will-change

```css
will-change: transform;
```

它提前告诉浏览器该层会频繁改变 transform，浏览器可以提前准备合成资源，减少第一次滚动时临时建层的成本。

`will-change` 不能大范围使用。每个项目都设置可能创建大量图层、增加内存和 GPU 纹理开销。本实现只给唯一的可见项目容器设置。

#### 6. translate3d 不解决什么

- 不会减少 DOM 数量，DOM 数量由虚拟化区间决定。
- 不会减少 Vue 响应式更新，更新次数由 rAF 合帧控制。
- 不会解决不定高度定位，定位依赖高度缓存和 Fenwick Tree。
- 不会突破浏览器最大元素高度，百万级列表需要物理、逻辑坐标压缩。
- 不保证完全没有 paint，阴影、滤镜和复杂子元素仍可能产生绘制成本。

#### 7. 百万级列表为什么不能直接使用逻辑 offsetY

当逻辑高度达到数亿像素时，直接把数亿像素写入 transform 可能遇到浏览器坐标范围和浮点精度问题。本项目先把滚动条限制在安全物理高度，再计算可见层的局部补偿位置：

```text
renderOffsetY =
    physicalScrollTop
    - (logicalScrollTop - logicalRangeOffset)
```

最终写入 translate3d 的仍是安全物理坐标附近的位置，而 Fenwick Tree 保留完整逻辑偏移。

## 不定高度实现与问题处理

### 1. 为什么需要预估高度

项目进入 DOM 前无法知道最终高度，但滚动条必须先有完整高度。因此所有未测量项目先使用 `estimatedItemHeight`，本示例根据 3 到 10 行内容的平均分布预估为 205px。

渲染区间内的项目挂载后，由一个 `ResizeObserver` 获取真实高度，再调用 `updateItemHeight(index, height)` 写回核心模型。高度差小于 0.5px 时忽略，避免字体和浏览器小数布局造成无意义更新。

不要把 Canvas 文本测量当成通用答案。Canvas 无法完整复现 DOM 的字体回退、CSS 换行、富文本、图片和异步组件。推荐优先级是：

1. 产品层面约束卡片高度。
2. 图片预留宽高，文本限制行数。
3. 按内容类型提供多个可靠预估高度。
4. 最后使用 ResizeObserver 修正已渲染项目。

### 2. 为什么使用 Fenwick Tree

不定高列表需要支持三种操作：更新单项高度、查询某项顶部累计偏移、根据 `scrollTop` 查找当前项目。普通前缀和查询是 O(1)，但任意一项高度变化都要修改后续所有前缀值，更新成本为 O(n)。

`DynamicHeightVirtualizer` 使用高度数组加 Fenwick Tree：

| 操作 | 复杂度 |
| --- | --- |
| 初始化高度树 | O(n) |
| 修正单项高度 | O(log n) |
| 查询累计偏移 | O(log n) |
| 根据滚动位置定位项目 | O(log n) |

这使高度修正成本与总数据量基本解耦。视口只渲染附近项目，因此 ResizeObserver 也只观察几十个节点，而不是观察全部数据。

#### Fenwick Tree 的数据结构原理

Fenwick Tree 也叫 Binary Indexed Tree（二进制索引树）。它不是保存每个完整前缀和，而是让每个树节点保存一段长度为 `lowbit(index)` 的区间和：

```text
lowbit(index) = index & -index

tree[index] 管理区间：
[index - lowbit(index) + 1, index]
```

树使用从 1 开始的索引，因为索引 0 的 `lowbit(0)` 仍为 0，无法向父节点移动。假设 8 个项目高度为：

```text
项目索引       1  2  3  4  5  6  7  8
项目高度       3  2  5  4  1  6  2  3
Fenwick 节点   3  5  5 14  1  7  2 26
```

每个节点覆盖范围如下：

| 节点 | lowbit | 保存的区间 | 保存值 |
| --- | --- | --- | --- |
| `tree[1]` | 1 | `[1, 1]` | 3 |
| `tree[2]` | 2 | `[1, 2]` | 5 |
| `tree[3]` | 1 | `[3, 3]` | 5 |
| `tree[4]` | 4 | `[1, 4]` | 14 |
| `tree[6]` | 2 | `[5, 6]` | 7 |
| `tree[8]` | 8 | `[1, 8]` | 26 |

#### 单项高度更新

假设第 3 项真实高度比预估高度多 2px，只有包含第 3 项的节点需要增加 2。更新从 `index = 3` 开始，每次执行：

```text
index += lowbit(index)
```

访问路径为：

```text
3 -> 4 -> 8
```

因此只更新 `tree[3]`、`tree[4]`、`tree[8]`。每次索引都会跳到覆盖范围更大的父节点，最多访问树高数量的节点，所以复杂度是 O(log n)。

#### 前缀高度查询

查询前 6 项累计高度时，从 `index = 6` 开始，每次执行：

```text
sum += tree[index]
index -= lowbit(index)
```

访问路径为：

```text
6 -> 4 -> 0
```

`tree[6]` 保存 `[5, 6]`，`tree[4]` 保存 `[1, 4]`，两个互不重叠区间正好组成 `[1, 6]`，结果为 `7 + 14 = 21`。每次都移除当前索引最低有效二进制位，因此最多执行 O(log n) 次。

#### 根据 scrollTop 定位项目

普通二分查找可以反复调用前缀查询，整体为 O(log² n)。当前实现使用 Fenwick Tree 的 binary lifting（二进制提升）直接查找“累计高度不超过目标 offset 的最大索引”。

算法从不超过项目数量的最高 2 次幂开始，例如 8；尝试将当前索引加上该步长：

1. 如果候选节点没有越界，且候选累计高度仍小于等于 `scrollTop`，接受这次跳跃。
2. 步长右移一位，例如 `8 -> 4 -> 2 -> 1`。
3. 最终得到目标位置之前完整项目的数量，也就是包含该 offset 项目的定位基础。

每个二进制位只检查一次，所以定位仍是 O(log n)。项目数组使用 0-based 索引，而 Fenwick Tree 使用 1-based 索引，写入高度时需要 `itemIndex + 1`，返回渲染索引时需要注意开闭区间。

### 3. 高度修正为什么会跳动

如果视口上方项目从预估高度修正为真实高度，它下面所有内容的逻辑位置都会变化。只更新总高度会让用户正在阅读的内容突然上移或下移。

修正前先记录项目底部偏移。如果项目完全位于当前逻辑 `scrollTop` 上方，则把高度差 `delta` 累加到滚动位置：

```text
newLogicalScrollTop = oldLogicalScrollTop + delta
```

这样内容在视口中的相对位置保持不变。位于视口内部或下方的项目不补偿，因为它们不会改变当前锚点之前的累计高度。

### 4. 为什么 50 万、100 万条曾经只能显示约 16 万条

浏览器对单个元素的可布局高度存在硬上限。Chromium 常见上限约为 33,554,432px。当前预估高度为 205px，因此未经处理时的最大可滚动数据量约为：

```text
33,554,432 / 205 ≈ 163,680
```

这就是切换到 50 万或 100 万条后，滚动条仍只能到约 16 万条的原因。数据已经生成，Fenwick Tree 也保存了完整高度，真正被截断的是负责撑开滚动条的 DOM 占位元素。

单纯继续增大 `height` 无法解决，因为浏览器会静默钳制布局尺寸。缩小所有行的视觉高度也不正确，会破坏真实排版。

### 5. 物理坐标与逻辑坐标压缩

解决方案是将滚动系统拆成两套坐标：

- 物理坐标：浏览器真实的 `scrollTop` 和占位元素高度，最大限制为安全值 8,000,000px。
- 逻辑坐标：完整数据的累计真实高度，可以达到数亿像素，只存在于 JavaScript 和 Fenwick Tree 中。

压缩比例使用“可滚动距离”计算，而不是直接用两个总高度相除，这样滚动条顶部和底部才能精确对应：

```text
scrollScale =
    (logicalTotalHeight - viewportHeight)
    / (physicalTotalHeight - viewportHeight)

logicalScrollTop = physicalScrollTop * scrollScale
```

物理占位高度为：

```text
physicalTotalHeight = min(logicalTotalHeight, 8,000,000)
```

虚拟区间始终使用逻辑 `scrollTop` 查询，因此能够定位 100 万条数据中的任意项目。实际 DOM 节点不能直接按比例压缩，否则文字和卡片高度也会失真。渲染块使用以下偏移保持当前逻辑锚点：

```text
renderOffsetY =
    physicalScrollTop - (logicalScrollTop - logicalRangeOffset)
```

### 6. 底部为什么会抖动

在列表底部，新渲染项目仍可能从预估高度修正为真实高度，导致逻辑总高度和压缩比例变化。如果继续使用旧逻辑位置反算物理位置，浏览器会把结果钳制到新的滚动底部，引发以下循环：

```text
测量高度
  -> 总高度和比例变化
  -> 写入新的 scrollTop
  -> 虚拟区间变化
  -> 新节点再次测量
```

高压缩比例还会放大边界误差。例如物理坐标变化 0.5px，在 25x 压缩下相当于逻辑坐标变化 12.5px，足以引起可见区间反复切换。macOS 触控板的边界回弹会进一步放大这个问题。

当前实现使用三层稳定策略：

1. 测量前判断是否已经位于物理底部。如果是，高度修正后直接锁定新的物理底部和逻辑底部，而不是保留旧位置。
2. 距顶部或底部不足 1px 时，不再做比例乘法，直接映射为精确的逻辑边界。
3. 已到边界后继续向外滚动时阻止无效 `wheel` 输入，并通过 `overscroll-behavior: contain` 阻止滚动链和页面回弹。

程序写回 DOM 前还会比较目标位置，只有差值超过 0.5px 才设置 `scrollTop`，减少程序化滚动产生的额外 `scroll` 事件。

### 7. 切换数据规模时必须重置什么

不定高页面不能只替换数据数组。不同规模的数据必须同步重置：

- 断开旧列表项的 ResizeObserver。
- 清空节点引用和已测量索引。
- 创建新的 `DynamicHeightVirtualizer` 和高度树。
- 重置物理、逻辑滚动位置及响应式版本号。
- Vue 更新 DOM 后重新观察首屏节点。

如果沿用旧高度缓存，同一个索引可能继承上一批数据的高度，导致总高度、定位结果和滚动补偿全部错误。

## 何时使用 Worker、分页或成熟库

| 场景 | 推荐方案 |
| --- | --- |
| 固定高度，10 万条以内 | 本页面的 O(1) 虚拟化 |
| 动态高度、吸顶、反向列表 | TanStack Virtual 等成熟库 |
| 本地重排序或过滤耗时超过一帧 | Worker 计算，主线程只接收结果索引 |
| 数据来自服务端且超过约 1,000 条 | 服务端分页、搜索、游标加载 |
| 百万级本地分析 | Worker + 分块数据，必要时使用 WASM 数据引擎 |

虚拟列表只解决渲染数量，不解决数据传输、内存占用和业务可发现性。用户通常无法有效浏览几十万条记录，因此生产系统应优先提供搜索、筛选和服务端分页。

## 验证标准

在 Chrome Performance 面板中录制快速滚动，重点检查：主线程是否持续出现超过 50ms 的 Long Task；DOM 节点是否随数据量增长；滚动期间是否出现强制同步布局；内存是否在往返滚动后持续上升。

本页面应满足：

- 100,000、500,000 和 1,000,000 条模式下 DOM 数量基本恒定。
- 500,000 和 1,000,000 条能够滚动到最后一条，不受浏览器最大元素高度限制。
- 到达底部后继续滚轮输入不会导致列表反复抖动。
- 不定高项目完成测量后，用户当前看到的内容不会明显跳动。
- 切换数据规模或卸载页面后没有遗留动画帧、观察器和旧高度缓存。
- 固定高列表的选择状态在项目滚出视口再滚回后仍然存在。
